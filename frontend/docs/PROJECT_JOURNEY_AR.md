# رحلة مشروع BATTECHNO-LMS (الواجهة الأمامية) — من البداية إلى الآن

هذا المستند يلخّص **بنية المشروع** و**ما بُني على مراحل** و**أين وصلنا اليوم**، وليس تفاصيل i18n فقط (للتفاصيل التقنية للترجمة راجع [`I18N_AND_LOCALE_AR.md`](./I18N_AND_LOCALE_AR.md)).

---

## 1. ما هو المشروع؟

- **اسم المنتج:** BATTECHNO LMS (نظام إدارة تعلّم لمؤسسة تقنية التعليم).
- **نطاق هذا المستودع (الجزء المعروض):** تطبيق **ويب أمامي (SPA)** مبني بـ **React + Vite**.
- **الهدف الوظيفي:** واجهات لأدوار متعددة (إدارة، مدرب، طالب، مراجع جامعي) مع **توجيه حسب الدور**، **صلاحيات واجهة**، وصفحات **CRUD** ولوحات تشغيل — جاهزة هيكلياً للربط لاحقاً بواجهة خلفية (API).

---

## 2. التأسيس التقني (البداية)

| العنصر | الاختيار |
|--------|-----------|
| بناء وتشغيل | **Vite 5** |
| واجهة | **React 18** |
| التوجيه | **react-router-dom v6** |
| طلبات/كاش (جاهز للتوسع) | **TanStack React Query** + **axios** |
| التحقق من المدخلات | **zod** |
| الأيقونات | **lucide-react** |
| الأنماط | **SCSS** (متغيرات، تخطيطات، صفحات) |
| اللغات (المرحلة الحالية) | **i18next** + **react-i18next** |

**نقطة الدخول:** `src/main.jsx` → يحمّل الأنماط العامة، ثم **تهيئة i18n**، ثم يصيّر `<App />`.

**هيكل عالي المستوى:**

```
src/
  App.jsx
  main.jsx
  app/
    providers/     # مزوّدو السياق (استعلام، ثيم، لغة، راوتر، مصادقة)
    router/        # كل المسارات
  layouts/         # أغلفة: إدارة، مدرب، طالب، مراجع، تسجيل دخول
  pages/           # صفحات حسب الدور والوحدة
  components/      # مشتركة، جداول، نماذج، تنقل
  features/        # auth, locale, theme
  constants/       # أدوار، تنقل، صلاحيات واجهة
  utils/             # مساعدات، تخزين، بوابات، تنسيق
  i18n/              # ترجمة وملفات locales
  assets/styles/     # SCSS
```

---

## 3. طبقة التطبيق (AppProviders)

ملف: `src/app/providers/index.jsx`

- **QueryClientProvider** — جاهز لربط API لاحقاً.
- **ThemeProvider** — ثيم فاتح/داكن (مخزّن محلياً).
- **LocaleProvider** — يزامن مع **لغة i18next** (اختيار المستخدم).
- **BrowserRouter** — التوجيه.
- **AuthProvider** — حالة المستخدم والجلسة (حالياً نمط تطوير/محاكاة تسجيل دخول).

---

## 4. المصادقة والبوابات (منذ التصميم الأول)

- مسارات تسجيل دخول منفصلة حسب **البوابة:**
  - `/login/admin`, `/login/instructor`, `/login/student`, `/login/reviewer`
  - صفحة `/login` تعيد التوجيه حسب **اسم المضيف (subdomain)** إن وُجد (انظر `utils/portal.js`).
- بعد تسجيل الدخول: **`ProtectedRoute`** يحمي المسارات التي تحتاج مستخدماً.
- **`RoleBasedRoute`** يقيّد المسارات حسب **الدور** (مثلاً فقط أدوار الإدارة لمسار `/admin/*`).

**الأدوار المعرفة:** `constants/roles.js` — مثل `super_admin`, `program_admin`, `academic_admin`, `qa_officer`, `instructor`, `student`, `university_reviewer`.

---

## 5. الصلاحيات على مستوى الواجهة

- ملف `constants/permissions.js` يعرّف **صلاحيات واجهة** (UI).
- **`RoleShellPermissionOutlet`** + **`hasUiPermission`** يحدّدان ما يظهر في القائمة الجانبية للمدرب/الطالب/المراجع حسب الدور والصلاحية.
- الإدارة تستخدم **قائمة مجمّعة** مع **تصفية حسب الدور الإداري** (انظر `adminNavigation.js`).

---

## 6. التخطيط الموحّد (Dashboard shell)

- **`BaseDashboardLayout`:** هيدر، قائمة جانبية، محتوى، فوتر — مشترك بين الإدارة، المدرب، الطالب، المراجع.
- كل من `AdminLayout`, `InstructorLayout`, … يغلّف نفس القاعدة مع مسارات مختلفة.

---

## 7. ما بُني في مسار الإدارة (`/admin`)

تم بناء **هيكل ومسارات وصفحات** تغطي على سبيل المثال:

- **لوحة**، **مستخدمون** (قائمة + إنشاء + عرض + تعديل)، **أدوار وصلاحيات** (صفحة).
- **جامعات، مسارات، شهادات مصغرة، مخرجات تعلم، دفعات، محتوى**.
- **جلسات، حضور، تقييمات** (مع CRUD للتقييمات في الإدارة)، **rubrics، تسليمات، درجات، أدلة**.
- **جودة، مراجعات جودة، إجراءات تصحيحية، طلبة متعثرون، مخاطر، نزاهة**.
- **طلبات اعتراف أكاديمي** (قائمة + CRUD)، **شهادات، تقارير، سجل تدقيق، إعدادات**.
- أي مسار إداري غير معرّف صراحةً يقع على **`ModulePlaceholderPage`** (صفحة مؤقتة موحّدة).

---

## 8. مسار المدرب (`/instructor`)

لوحة، دفعات، جلسات، حضور، تقييمات (مع إنشاء/تعديل للمدرب)، تسليمات، درجات، أدلة، طلبة متعثرون — مع **`RoleShellPermissionOutlet`** للعناصر حسب الصلاحية. المسارات غير المعروفة → placeholder.

---

## 9. مسار الطالب (`/student`)

لوحة، برامج/شهادات مسجّل بها، محتوى، جلسات، حضور، تقييمات، تسليمات، درجات، شهادة رقمية — و placeholder للباقي.

---

## 10. مسار المراجع الجامعي (`/reviewer`)

لوحة، طلبات اعتراف، تقارير جامعة، أدلة، شهادات مرتبطة — و placeholder.

---

## 11. المكوّنات والأنماط المشتركة

- جداول (`DataTable`)، نماذج (`FormInput`, `FormSelect`, …)، أزرار، بطاقات، حالات فارغة، تحميل، ترويسات صفحات.
- تنقل: `AdminSidebar`, شريط علوي `AppNavbar`, جرس إشعارات، قائمة مستخدم.
- **SCSS** منظّم: متغيرات، تخطيطات، صفحات حسب المنطقة (إدارة، مدرب، طالب، مراجع).
- **ثيم** فاتح/داكن مع تخزين التفضيل.

---

## 12. البيانات والواجهة الخلفية

- كثير من الصفحات تعتمد على **بيانات وهمية (mocks)** أو نصوص ثابتة لغرض العرض والتصميم إلى حين ربط **API** حقيقي.
- **React Query** جاهز لاستبدال الاستدعاءات لاحقاً.

---

## 13. المرحلة الأخيرة: التعدد اللغوي (أين وصلنا الآن)

تم اعتماد **i18next** كمصدر رئيسي للنصوص، مع:

- **العربية افتراضياً**، **الإنجليزية**، حفظ اللغة، **RTL/LTR** على المستند.
- **مبدل لغة** في الهيدر، وترجمة **القائمة الجانبية والعناوين** عبر مفاتيح في `navigation` ووحدات أخرى.
- ملفات JSON تحت `src/i18n/locales/{ar,en}/`.
- أدوات مساعدة: `utils/locale.js`, `utils/format.js`.

**التفاصيل الكاملة للترجمة والملفات:** راجع **[`I18N_AND_LOCALE_AR.md`](./I18N_AND_LOCALE_AR.md)**.

**ما يزال عملاً مستمرّاً:** ترحيل **كل** نصوص الصفحات الداخلية من أي أسلوب قديم (`useTr` / نص عربي ثابت في JSX) إلى **`useTranslation` + مفاتيح** حتى لا يبقى خلط لغات في واجهة واحدة.

## 14. ملخص «من البداية إلى الآن»

| المرحلة | المحتوى |
|--------|---------|
| **1 — التأسيس** | React + Vite، هيكل مجلدات، SCSS، مكوّنات أساسية |
| **2 — التوجيه والأدوار** | 4 بوابات، مسارات `/admin`, `/instructor`, `/student`, `/reviewer`، حماية المسارات |
| **3 — الإدارة والوحدات** | CRUD ولوحات لمعظم وحدات LMS الإدارية + placeholder للمسارات الناقصة |
| **4 — الصلاحيات والتجربة** | صلاحيات واجهة، قوائم ديناميكية، ثيم، بوابة subdomain |
| **5 — التعدد اللغوي** | i18n، RTL/LTR، مفاتيح ترجمة، غلاف التطبيق والتنقل مترجَمين؛ باقي الصفحات قيد التوحيد |

---

## 15. خطوات تالية مقترحة (خارج نطاق «ما تم»)

- ربط **API** حقيقي واستبدال الـ mocks.
- إكمال **ترجمة** جميع الشاشات بنفس أسلوب المفاتيح.
- اختبارات (وحدة / تكامل) حسب احتياج الفريق.
- توثيق API وبيئة التشغيل في مستودع أوسع إن وُجد backend منفصل.
